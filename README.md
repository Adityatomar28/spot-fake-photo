# sales-code-ai — Screen vs. Real Photo Classifier

Detects whether a submitted product photo is a **real photo** or a **photo of a screen** (a recaptured/forwarded image), using classical computer-vision features and a small logistic regression / SVM classifier — no deep learning, no GPU required.

## How it works

1. **Feature extraction** (`features.py`) — pulls 16 hand-engineered features out of each image:
   - **FFT spectral features** — radial energy distribution in the frequency domain, tuned to catch moiré/aliasing patterns typical of screen recaptures.
   - **Angular FFT features** — directional concentration of spectral energy (screen moiré tends to be directional, not just radially elevated).
   - **LBP entropy** — local binary pattern texture randomness.
   - **Glare / highlight features** — clipped highlights and large bright blobs (common in screen photos due to reflections).
   - **Saturation stats** — mean/std of color saturation.
   - **Sharpness grid** — Laplacian variance across a 4×4 grid, capturing focus uniformity.
   - **Edge line density** — Hough-line count, normalized by area.
   - **RGB channel mismatch** — gradient differences across color channels.
   - **Block artifact score** — proxy for 8×8 JPEG block boundaries, indicating recompression (e.g. WhatsApp-forwarded images).

   Spectral/texture features are computed on a **center crop (65% of the frame)** rather than the full image, so a busy patterned background near the frame edges doesn't get mistaken for screen moiré.

2. **Training** (`train.py`) — reads images from `real/` (label 0) and `screen/` (label 1), extracts features, standardizes them, then grid-searches:
   - Logistic Regression (L1 / L2 penalty, multiple `C` values)
   - RBF-kernel SVM (multiple `C` / `gamma` values)

   The best model by 5-fold stratified cross-validation accuracy is kept, fit on the full dataset, and saved to `model.pkl` along with the fitted scaler, feature names, and a suggested decision threshold (tuned to favor precision — i.e. fewer false "screen" flags — at ≤5% false positive rate).

3. **Prediction** (`predict.py`) — loads `model.pkl` and classifies new images.

## Folder structure

```
sales-code-ai/
├── real/              # training images: genuine product photos
├── screen/            # training images: photos of a screen/display
├── features.py        # feature extraction
├── train.py            # training + model selection
├── predict.py          # inference on new images
├── model.pkl            # saved scaler + classifier (generated by train.py)
└── demo_server.py     # demo/serving script
```

## Usage

### Train

Put your labeled images in `real/` and `screen/` (any of `.jpg`, `.jpeg`, `.png`, case-insensitive), then:

```bash
python train.py
```

This prints:
- CV accuracy for every hyperparameter combination tried (logistic regression + SVM)
- The best model and its parameters
- A confusion matrix
- **A list of misclassified image filenames** — useful for spotting labeling errors or systematic failure patterns rather than guessing blindly
- The suggested decision threshold
- Feature-extraction latency benchmark

### Predict

```bash
python predict.py path/to/image.jpg
```

## Current performance

As of the latest training run: **~94.3% 5-fold cross-validated accuracy** on 140 labeled images (73 real, 67 screen).

### Known failure patterns

- **WhatsApp-compressed screen photos** — WhatsApp's aggressive recompression can smooth over the moiré signal the model relies on, occasionally causing screen photos to be misclassified as real. Mitigation: include more WhatsApp-sourced examples in `screen/`, and/or strengthen the block-artifact feature.
- **Real photos with busy/patterned backgrounds** — a strongly patterned backdrop (e.g. fabric with dense repeating motifs) can produce screen-like directional spectral energy. The center-crop preprocessing reduces this, but training on more varied backgrounds (plain + patterned) for both classes further improves robustness.

## Improving accuracy further

1. Run `train.py` and review the misclassified filenames it prints — open those images and look for a pattern before changing any code.
2. Add more training images that resemble the failure patterns above — small-dataset classifiers usually gain more from a handful of well-chosen new examples than from extra feature engineering.
3. Keep the `real/` and `screen/` folders balanced in style (lighting, backgrounds, compression source) so the model learns to distinguish *screen recapture artifacts*, not incidental differences between your two photo sources.
4. Consider nested cross-validation if you need an unbiased accuracy estimate — the current grid search picks the best model using the same folds it reports accuracy on, which is slightly optimistic.

## Notes

- All features are computed at ~80 ms/image on CPU — no GPU needed.
- `model.pkl` contains everything needed for inference: scaler, classifier, feature names, threshold, and the CV accuracy at time of training.
